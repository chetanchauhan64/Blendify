## Migration Strategy

### Development Workflow
```bash
# 1. Make changes to prisma/schema.prisma
# 2. Create and apply migration:
npm run db:migrate
# Enter migration name when prompted, e.g.: "add_loyalty_table"

# 3. View DB in Prisma Studio:
npm run db:studio

# 4. Seed development data:
npm run db:seed
```

### Production Deployment
```bash
# Apply all pending migrations (no prompts):
npm run db:migrate:prod

# !! NEVER run db:push or db:reset in production !!
```

### Migration Naming Convention
```
YYYYMMDDHHMMSS_<description>
e.g.: 20240615120000_initial_schema
      20240620090000_add_loyalty_tiers
      20240701140000_add_analytics_events
```

## Key Design Decisions

### 1. Price Storage
- All prices stored in **USD** in the `basePrice` field
- `RegionPrice` table overrides per-country with local currency
- `Order.currencyCode` captures the currency used at checkout
- `Order.totalUsd` and `Order.exchangeRateUsed` enable multi-currency reporting

### 2. Inventory Safety
- Stock changes are **always** done via `InventoryLog` + atomic transaction
- `InventoryLog` is append-only — never delete or update records
- Every movement has an `action` enum, `reference` (e.g. orderId), and optional `performedById`
- `ProductVariant.lowStockThreshold` drives low-stock alerts

### 3. Order Lifecycle
- Status transitions enforced in `OrderService.cancelOrder()` — e.g. can't cancel a `SHIPPED` order
- Every status change creates an `OrderTimeline` entry (public or admin-only)
- `OrderTimeline.isPublic = false` entries visible only to admin

### 4. Soft Delete Strategy
- No `deletedAt` columns — use `isActive: false` or `status: ARCHIVED`
- Keeps all relations intact for reporting and audit trails
- Products: `ProductStatus.ARCHIVED`
- Variants: `ProductVariant.isActive = false`
- Users: `User.isActive = false`

### 5. Guest Checkout
- `Order.userId` is nullable — `Order.guestEmail` captures guest identity
- Guest carts use `Cart.sessionId` instead of `Cart.userId`

### 6. Multi-Currency Architecture
- `Currency` table with `exchangeRate` relative to USD
- `Country` → `Currency` via `currencyCode` FK
- `RegionPrice` allows per-product, per-country price overrides
- Exchange rates updateable via `CurrencyRepository.updateExchangeRates()`

## Connecting to PostgreSQL

### Local Development (Recommended: Docker)
```bash
docker run --name blendify-db \
  -e POSTGRES_PASSWORD=password \
  -e POSTGRES_DB=blendify_db \
  -p 5432:5432 \
  -d postgres:15

# Then set in .env:
DATABASE_URL="postgresql://postgres:password@localhost:5432/blendify_db"
DIRECT_URL="postgresql://postgres:password@localhost:5432/blendify_db"
```

### Production (Supabase / Railway / Neon)
```bash
# Use the connection pooler URL for DATABASE_URL
# Use the direct URL for DIRECT_URL (for migrations)
DATABASE_URL="postgresql://...pooler.supabase.com:6543/postgres?pgbouncer=true"
DIRECT_URL="postgresql://...direct.supabase.com:5432/postgres"
```
